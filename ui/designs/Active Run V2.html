Artifacts placement · 3 options

Where to surface spec, plan, and other run-generated files.

The artifacts (.feature spec, plan.md, possibly analysis reports later) need to stay reachable for the full life of the run — during the gate, during autonomous execution, and after the PR is merged. Below are three placements that keep them one click away from the active-run screen, ranked roughly from "most discoverable" to "most ambient". My recommendation is bolded in option α.

What counts as an artifact
BDD spec file outbox-dispatcher.feature
Execution plan plan.md
Jira story link (created by jira-agent)
Later: review notes, ADO PR link, commit SHA
Requirements
Reachable during the gate AND after it
Glanceable count + last-updated
Click → side panel (don't navigate away)
Doesn't compete with the decision queue
Option α · recommended

Topbar pill with dropdown

A persistent Artifacts · n pill on the right side of the topbar, beside the run-control cluster. Opens an inline dropdown listing each artifact with a label, who produced it, and a quick metric. Click an item → opens in a side drawer; doesn't navigate away from the active run.

Anchored — present in every state of the run
Counts (3) read as a glanceable status
Topbar already houses run-level metadata, fits the model
Dropdown design needs to scale to 6+ artifacts post-merge
