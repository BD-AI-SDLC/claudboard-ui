Overview — three project setups, plus first-run

The Overview surface should adapt to the project's setup type. Universal blocks (active runs, gates, activity, throughput) stay; scenario-specific blocks (repo list, module list, edge map) appear only where they make sense. The workspace switcher at the top of the sidebar is the single entry point for adding or switching workspaces — first-run is its empty state.

Scenario 1 · Monolith

One repo, one git history

Smallest case. No "repositories" list — there's only one. Recent runs is the main content card; in-flight runs sit at the top. If claudboard's analysis surfaces natural subareas (controllers / persistence / messaging in a Spring app, for example), they show as a compact modules list — purely informational, not a directory of separate units.

Scenario 4 · First run

No workspaces yet

The sidebar's workspace switcher is the empty state. Three import paths: open a local folder (claudboard auto-detects the type), clone from Git URL, or continue from .claude on disk — surfaces only if claudboard finds an existing setup file in your home or common workspace dirs. The detected card has priority styling so power users get the fast path.

Detection logic
.git in selected folder → monolith or monorepo (workspaces detected from package.json / pnpm-workspace.yaml / nx.json)
no .git but child folders each have .git → multi-repo workspace
.claude/ at root → import existing setup
Subsequent flow
After import, jump to Project Health setup chain
.claude/ found → mark steps as imported
Workspace appears in switcher with type tag
