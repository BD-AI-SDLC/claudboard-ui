Active run · v3 — run states & artifact viewer

Three states of the same screen, plus the artifact-viewing flow. Gate shows the human-approval pause we've been iterating on — now with clickable spec/plan cards inside the decision box. Running is what 90% of the time looks like: no decision needed, agents producing under their own steam, tasks ticking off in the rail, telemetry climbing. Clarification is for the in-between case — a lightweight question the run can't quite resolve on its own, presented inline without halting the whole pipeline. Artifact viewer shows what happens when you click a file from any of those states. Right rail is now telemetry first, then tasks.

Paused at the human checkpoint between phase 1 and phase 2.

State 3 · Clarification

A small, in-flight question — not a full gate.

For decisions the run can't quite resolve on its own but that don't warrant pausing the entire workflow. Renders as a compact violet card between the pipeline and the now-working card, with 2–4 quick-reply options and a "decide later" affordance. Other tasks in the same checkpoint that don't depend on the answer keep running — the run only really stalls when it hits something downstream of the choice.

Visual weight vs. gate
Single-row card, violet accent (not amber)
Status bar stays green-ish; only the question is highlighted
Quick-reply chips with keyboard shortcuts
"Decide later" defers to a sensible default after timeout
Examples this fits
"Use tenantId or compositeKey for partition?"
"PR target branch — main or develop?"
"Pin lib X to v3 or stay on v2?"

Quick question from implementation-agent · checkpoint 3 · waiting 42s
Kafka partition key: use tenantId alone, or tenantId + aggregateId for stricter ordering inside a tenant?
Spec asks for FIFO ordering per tenant. tenantId alone satisfies it but allows interleaving across aggregates in one partition. The compound key narrows ordering to the aggregate but increases partition count. Other dispatchers in this repo use tenantId.

State 4 · Artifact viewer

Open the file inline — go back to the run when you're done.

Clicking any artifact (from the gate box, the topbar dropdown, or anywhere else) slides a viewer in over the run. The run is dimmed but still visible behind, so you don't lose context. A "← Back to run" button is always front and centre. Multiple artifacts you've opened become tabs at the top of the viewer, so you can flip between spec and plan without re-opening.

What's on this screen

Why a sheet, not a separate page

1# language: en
2Feature: Outbox dispatcher for entitlement events
3
4 As the meas.cloud.datahandler service
5 I want to publish entitlement-domain events to Kafka exactly-once
6 So that downstream services react in near-real-time
7
8 Background:
9 Given the dispatcher worker is scheduled every 500ms
10 And the Kafka topic "entitlement.events.v1" is reachable
11
12 Scenario: Happy path · single tenant publishes once
13 Given tenant "acme" has 3 pending outbox events
14 When the dispatcher tick runs
15 Then all 3 events are published to "entitlement.events.v1"
16 And they are marked PUBLISHED in the outbox collection
17 And no event is published a second time on a subsequent tick
18
19 Scenario: FIFO order preserved within a tenant
20 Given tenant "acme" has events A, B, C inserted in order
21 When the dispatcher tick runs
22 Then Kafka offsets for those events are in the same order
23
24 Scenario: Broker outage · events stay pending and retry
25 Given the Kafka broker is unreachable for 10s
26 When 3 ticks elapse
27 Then events remain PENDING with claim counter incremented
28 When the broker becomes reachable again
29 Then the next tick publishes all pending events in original order
