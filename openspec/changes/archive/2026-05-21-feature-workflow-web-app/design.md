## Context

The `feature-workflow` Claude Code skill is a per-repo orchestrator generated by the `claudboard-workflow` skill in the claudboard plugin. It coordinates eight sub-agents across seven phases (ticket → BDD spec → architect plan → human gate → branch → develop → commit → review → PR → finalize), and today it runs only inside the `claude` CLI. The bosch-workflow React mock at `/Users/LUP1BG/Documents/BoschProjects/bosch-workflow/project` defines the target visual surface in detail — five screens, a token-driven CSS system, and an event/phase/stream data model close to what a real server would emit.

This design covers the runtime spine that lets a Node-hosted UI drive that same skill end-to-end without forking it.

## Goals / Non-Goals

**Goals:**
- Drive the existing `feature-workflow` skill unchanged in shape — same `.claude/skills/feature-workflow/` files run inside our process, the workflow is the source of truth.
- Provide a single npx-distributable local app with a UI that visually matches the bosch-workflow design.
- Support monolith, monorepo, and multi-repo workspace topologies day one.
- Make the human gate a typed, structured exchange rather than prose parsing.
- Reuse the user's existing Claude Code MCP configuration for all auth secrets.
- Persist transcripts so runs are auditable; support user-initiated pause/resume during a live session.

**Non-Goals:**
- Authentication, multi-tenant, or any concept of "remote users."
- Cloud deployment, hosted SaaS, remote repo support.
- Crash recovery — if the server process dies mid-run, that run cannot be resumed.
- Replacing the `claude` CLI for any other skill — only `feature-workflow` and the claudboard prereqs are wrapped.
- Auto-migrating already-generated feature-workflow installations. Users re-run `/claudboard-workflow` after the template edit lands.

## Decisions

### D1. Anthropic Agent SDK over `claude` CLI subprocess

**Decision:** Embed `@anthropic-ai/claude-agent-sdk` in the Node server and call `query()` per run, pointing `cwd` at the target repo.

**Why:** The SDK gives us first-class hooks, in-process MCP server registration, typed message streams, and avoids spawning per-run child processes. Critically, it loads the target repo's `.claude/` exactly as Claude Code does, so the workflow runs *identically* to the CLI invocation.

**Alternatives considered:**
- *CLI subprocess (`claude -p ... --output-format stream-json`)*: Zero workflow changes, but tail-parsing stdout for state is fragile, and bridging the gate back into stdin is awkward. Rejected.
- *Rebuild on raw Anthropic Messages API*: Loses the "exact workflow" property. Months of reimplementation. Rejected.

### D2. In-process MCP server for phase/gate events, not Bash sentinels

**Decision:** Register an in-process MCP server via `createSdkMcpServer({ name: 'bosch', tools: { phase_start, phase_complete, checkpoint_start, checkpoint_complete, agent_start, agent_complete, gate_request } })`. The generated `feature-workflow/SKILL.md` calls these typed tools at phase/checkpoint boundaries and at the Phase 1d approval gate.

**Why:** Typed payloads, no string parsing, no subprocess. The `gate_request` handler `await`s a deferred that the WS endpoint resolves when the UI replies — the model continues seamlessly without ever knowing a human was involved.

**Alternatives considered:**
- *Bash sentinel commands (`bosch-event phase-start --num 1 --title "..."`) + `PreToolUse` hook*: Works with no SDK extension surface, but regex parsing of CLI args is brittle and gate payloads (spec text, plan structure) would need temp files. Rejected.
- *Reusing the Notification/Stop hooks*: Too coarse — they don't carry per-event payloads. Rejected.

### D3. Three repo topologies normalized to `(target, scope?, workspaceRoot?)`

**Decision:** Every run is parameterized by three fields:
- `target`: absolute path used as the SDK `cwd`. Always a single repo.
- `scope`: optional sub-path appended to the `/start-feature` prompt to constrain a monorepo run.
- `workspaceRoot`: optional parent directory whose shared `.claude/` is read alongside `target/.claude/` (used by the workspace-link variant of claudboard).

**Why:** Collapses three UX modes to one server contract. The UI picks which fields to surface; the run-driver doesn't need topology-specific branches.

### D4. Reuse `~/.claude/` MCP configuration, do not build a secret store

**Decision:** Run the Agent SDK with the user's existing Claude Code MCP config so JIRA/ADO/Atlassian creds flow in transparently. Startup precondition: Claude Code must be installed and at least one MCP server must be configured.

**Why:** Building a secret store would mean a first-run wizard, key-rotation UX, and a second source of truth for credentials the user already manages. Rejected.

**Trade-off:** Users without Claude Code installed get a clear precondition failure and must install it first. This is acceptable for v1; the target audience is teams already using claudboard.

### D5. Pause/resume in-session only, via held async iterator

**Decision:** User-initiated pause is implemented by holding the SDK message iterator at the next message boundary using a deferred promise. Resume releases the deferred. SKILL-initiated gates use the same pattern via `gate_request`.

**Why:** Both pause types are just "the model produced output and the server is choosing not to feed the next turn yet." A held iterator captures that exactly. The Agent SDK's `resume: sessionId` exists but adds the implication of cross-process recovery — which we explicitly don't want.

**Trade-off:** Server crash kills the run irrecoverably. We accept this for v1; transcripts on disk are sufficient forensics.

### D6. No UI libraries; styles.css is the visual source of truth

**Decision:** Real React + TypeScript app via Vite. Plain CSS, per-component `.css` file imported alongside its `.tsx`. No Tailwind, no shadcn, no Radix, no MUI, no CSS-in-JS, no CSS modules.

**Why:** The bosch-workflow design encodes the visual language precisely in plain CSS (1053 lines, custom tokens). UI libraries would impose their own component shapes that fight the design — exactly the "playing with UI" the user explicitly forbade. Plain CSS keeps the contract direct.

**Note:** The mock's CDN-React + in-browser Babel approach is *not* adopted. The mock is a design reference; the implementation uses a proper build pipeline so we ship prebuilt assets via npx.

### D7. Workflow-instrumentation as a separate capability

**Decision:** The edit to `claudboard-workflow`'s SKILL template lives in its own spec (`workflow-instrumentation`) and is shipped via the claudboard plugin repo, not this one.

**Why:** It's a binary contract between this app and any repo whose feature-workflow it drives. Keeping it separate makes the cross-repo dependency explicit and lets us version it independently. Existing repos with a previously-generated feature-workflow will not be drivable until they re-run `/claudboard-workflow`.

### D8. SQLite for state, JSONL for transcripts

**Decision:** `~/.bosch-sdlc/state.db` (better-sqlite3) for workspaces, projects, runs (incl. status, timestamps, cost, token usage). `~/.bosch-sdlc/transcripts/<run-id>.jsonl` for per-run message streams.

**Why:** SQLite handles queries the dashboard needs (counts by status, recent runs). JSONL is append-only, easy to tail during a run, trivial to inspect post-mortem.

## Risks / Trade-offs

- **[Agent SDK version drift breaks the in-process MCP contract]** → Pin the SDK version in `package.json`; add a smoke test that asserts `createSdkMcpServer` is importable and `query()` accepts our options shape on startup.
- **[The claudboard-workflow template edit lands but downstream repos still have the old generated SKILL]** → The web app surfaces "Re-generate feature-workflow" as an explicit action in the Project screen when it detects an old SKILL signature (missing typed tool calls). Until re-generated, runs against that repo are rejected with a clear error.
- **[Held-iterator pause doesn't actually pause the model mid-turn]** → It pauses *between* messages, not mid-message. We accept that "Pause" may take a few seconds to take effect. The UI shows a "Pausing..." state until the next message boundary.
- **[`gate_request` deferred leaks if WS disconnects]** → Tie deferreds to the run record. If the WS reconnects within N seconds, re-emit the gate event; if not, the run remains in `paused-gate` and can be resumed from the dashboard.
- **[Reading `~/.claude/` cross-platform]** → Use `os.homedir()`; document the precondition explicitly; fail fast with a clear message if the directory is missing or has no MCP servers.
- **[Per-component CSS without modules risks global class collisions]** → Adopt a BEM-ish prefix convention per screen (`run-banner__title`, `gate-step__keyword`) ported from the existing styles.css naming. Lint a class-name prefix rule in CI.
- **[Visual parity is subjective]** → Define "parity" as: identical layout grid, identical typography stack (Geist + Geist Mono), identical color tokens, identical component shapes. A checklist screen-by-screen lives in the web-ui spec.

## Open Questions

- *Will any sub-agent (architect-agent, sdd-expert-agent, …) need to call the typed MCP tools too, or only the main orchestrator skill?* — Current plan: only the main SKILL. If sub-agent telemetry granularity becomes important, we add `agent_start`/`agent_complete` calls inside their .md templates too.
- *Multi-repo shared `.claude/` resolution order* — When `workspaceRoot/.claude/` and `target/.claude/` both define the same rule or skill, which wins? Default proposal: `target` overrides `workspaceRoot`, mirroring CSS specificity. Worth confirming once we test it.
- *Transcript size cap* — Long runs (hours, hundreds of messages) could produce multi-MB JSONL files. Acceptable for v1; revisit if it becomes a UI loading bottleneck.
