## Why

Feature runs cannot start from the bosch-sdlc UI on any project. The server's `skill-check.ts` rejects every SKILL.md generated by claudboard-workflow v3.8.1 because the template contains `AskUserQuestion` in documentation sections ("Halt mechanics") and as a fallback clarification mechanism. The check does a naive `content.includes('AskUserQuestion')` that cannot distinguish instructional use from documentary mentions.

The deeper issue: bosch-sdlc injects the `mcp__bosch__*` MCP server at runtime via the Agent SDK, but has no way to tell the feature-workflow skill to use it exclusively. The LLM might follow the `AskUserQuestion` path even when `mcp__bosch__*` tools are available, because the SKILL.md presents both as valid options.

This change adds a hard `--gate=mcp` flag to the prompt that bosch-sdlc sends, giving the skill an unambiguous instruction to use the MCP gate path. It also relaxes the over-aggressive validation to match the new contract.

Companion change in claudboard: `claude-repo-scan/openspec/changes/gate-mode-flag/`.

## What Changes

### 1. `prompt-builder.ts` — add `--gate=mcp` to invocation

`buildPrompt()` currently produces:

```
Start feature --autonomy=balanced: <user prompt>
```

It will produce:

```
Start feature --autonomy=balanced --gate=mcp: <user prompt>
```

This flag is parsed by the feature-workflow SKILL.md (see claudboard companion change) and forces the agent onto the `mcp__bosch__*` tool path for all gates, clarifications, and phase lifecycle calls. `AskUserQuestion` and end-of-turn halts are prohibited in this mode.

### 2. `skill-check.ts` — relax validation

Remove the `UNINSTRUMENTED_GATE_PATTERNS` blocklist and the loop that checks for `AskUserQuestion`, `Reply \`confirm\``, and `accept [Enter] or override`. These patterns are legitimate content in a dual-mode SKILL.md.

Keep the two remaining checks:
- File exists (`.claude/skills/feature-workflow/SKILL.md`)
- Contains `mcp__bosch__` references (proves the template knows about the MCP gate contract)

The hard guarantee shifts from "SKILL.md must not mention vanilla patterns" to "SKILL.md must support MCP mode" + "prompt forces MCP mode at invocation".

### 3. Update integration tests

Tests in `integration.test.ts` that assert 409 rejection for SKILLs containing `AskUserQuestion` must be updated. The new contract:
- SKILL.md missing → 409
- SKILL.md without `mcp__bosch__` → 409
- SKILL.md with both `mcp__bosch__` and `AskUserQuestion` → 201 (this is the dual-mode case)
- SKILL.md with only `AskUserQuestion` (no `mcp__bosch__`) → 409

## Capabilities

### Modified capabilities

- `run-lifecycle` — prompt builder adds `--gate=mcp` flag; skill validation relaxed to allow dual-mode SKILLs that contain both MCP and interactive gate paths.
