# bosch-sdlc

## What it is

`bosch-sdlc` is a web-based dashboard for driving the `feature-workflow` Claude Code skill via the Anthropic Agent SDK. It runs as a local Node server, watches Claude execute the SDLC workflow across seven phases (ticket → BDD spec → architect plan → human gate → branch → develop → commit → review → PR → finalize), lets the user review and approve the spec+plan gate from a browser, and streams all phase, checkpoint, and agent events live. The workflow itself lives unchanged in each target repo's `.claude/skills/feature-workflow/`; this app is a driver and UI layer on top of it.

## Who it's for

Teams using the `bosch-workflow` / `claude-repo-scan` toolchain — specifically any repo whose feature-workflow skill was generated by `/claudboard-workflow` — who want a visual dashboard instead of terminal output. Non-engineer stakeholders can use it to monitor and gate runs without touching the `claude` CLI.

## Prerequisites

1. **Claude Code** must be installed and available as `claude` on your PATH. Install it from [claude.ai/download](https://claude.ai/download). bosch-sdlc shells out to the CLI to run claudboard prereq commands (`/analyse`, `/generate`, `/claudboard-workflow`, `/refresh`, `/techdebt`).
2. **At least one MCP server** must be configured in `~/.claude/` (for JIRA, ADO, or any auth the feature workflow needs). `bosch-sdlc` reuses your existing Claude Code MCP configuration — no separate credential store.
3. **The claudboard plugin is installed automatically on first boot.** The dashboard shows a "Setting up bosch-sdlc…" card while the install runs in the background; once it disappears, you're ready to go. If the install fails (e.g. the marketplace is unreachable), the card shows the error and a Retry button. Re-attaching after upgrading Claude Code is unnecessary — the install is idempotent and only runs when the plugin is missing.
4. **A generated `feature-workflow` skill** must still exist in each target repo's `.claude/skills/feature-workflow/` for **Start Feature** to work. Generate or regenerate it by running `/claudboard-workflow` (the dashboard button) for that repo. Prereq commands themselves do not require this skill.

## Quickstart

```sh
npx bosch-sdlc
```

That is the only command. The server picks a free port, opens your browser automatically, and is ready to use. From there:

1. **Add a repo** — paste or browse to the target repo path on the Dashboard.
2. **Run setup** — open the Project screen; the SetupBanner highlights the next prereq (`Analyse` → `Generate` → `claudboard-workflow`). Click `▶ Run` and watch it run. Once the foundation is complete, Start Feature unlocks. **Interactive skills like `/claudboard-workflow` prompt for configuration inline** — a question card appears below the Foundation section; pick an option and click Submit to answer. The run resumes automatically.
3. **Kick off a feature** — go to Kickoff, enter a feature description, pick a clarification autonomy level (`autopilot` / `balanced` / `guided` / `manual`; the radio is pre-seeded from the repo's `config.clarify.defaultAutonomy`), and submit.
4. **Review the gate** — when the workflow pauses at Phase 1d, the Review Gate screen shows the generated spec and plan; approve or reject with change requests.

The run streams all events live in the Active Run screen (Pipeline / Stream / Telemetry panes) and persists a full transcript to `~/.bosch-sdlc/transcripts/<run-id>.jsonl`.

> **Note:** If the server process dies mid-run, that run cannot be resumed. Transcripts on disk are the forensic record.

## UI Theming

The UI respects the OS colour-scheme preference (`prefers-color-scheme`) on first load, defaulting to dark mode when the OS preference is unavailable. A moon/sun toggle in the sidebar footer lets the user override the theme for the current session; the override is not persisted across reloads.

## Development

Install dependencies from the repo root:

```sh
npm install
```

Then start the server and UI in separate terminals:

```sh
# Terminal 1 — API + WebSocket server (port 3001 by default)
npm run dev -w server

# Terminal 2 — Vite dev server for the React UI (proxies /api to server)
npm run dev -w ui
```

Full type-check, lint, and test across all workspaces:

```sh
npm run typecheck
npm run lint
npm run test
```

Build the publishable package (outputs to `server/dist/` + `ui/dist/`):

```sh
npm run build
```

## Launching claudboard skills from the dashboard

The dashboard includes a **Claudboard skills** panel (visible in the project/repo view once the foundation prereqs are complete). It provides one-click launchers for:

- **Analyse** — runs `/claudboard-analyse` on the repo; produces `.claude/reports/claudboard-analysis.md`
- **Generate** — runs `/claudboard-generate`; writes `CLAUDE.md`, rules, and skills from the analysis report
- **Workflow** — runs `/claudboard-workflow`; generates the `.claude/skills/feature-workflow/` skill for the repo

All inputs the skill would otherwise ask about interactively are collected upfront via a modal form, so the run executes fully non-interactively and streams events in real time to the live run view.

**Prerequisite:** the claudboard plugin must be installed at `~/.claude/plugins/marketplaces/claudboard/`. If it is missing, the launcher buttons are disabled with a tooltip. To install:

```sh
git clone <claudboard-repo> ~/.claude/plugins/marketplaces/claudboard
```

Free-form fields (e.g. Jira Cloud ID) support a **stub** checkbox — checking it submits `[TODO: FIELD_NAME]` in the generated config so you can run the skill and fill in real values later.

## Architecture

The server embeds `@anthropic-ai/claude-agent-sdk` and calls `query()` with `cwd` pointed at the target repo, which causes Claude to run the feature-workflow skill exactly as it would from the CLI. An in-process MCP server named `bosch` provides typed tools (`phase_start`, `gate_request`, etc.) that the skill calls to emit structured events; these are persisted to SQLite and broadcast over WebSocket to the React UI. State is stored in `~/.bosch-sdlc/state.db`; transcripts are appended to per-run JSONL files.
